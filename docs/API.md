# API & Frontend Conventions

How the web app (`colette-exercise-web`) talks to the backend, and the conventions you
must follow when building UI. The frontend is **React Router v7 in framework mode** with
**TypeScript**. You won't be asked to build the backend's transport from scratch — you
*will* be graded on following these conventions.

## The two repos

- `colette-exercise-api` — Elixir/Phoenix + Absinthe. Serves GraphQL at
  `POST http://localhost:4000/api`, playground at `/graphiql`.
- `colette-exercise-web` — React Router v7 framework-mode app. Fetches data from the API
  in route `loader`s/`action`s.

The backend SDL is the contract. It lives at
`colette-exercise-api/priv/schema.graphql` and is the source of truth for codegen.

## Authentication

The API uses a bearer token that **is the user id** (no JWT, this is an exercise).
Every request to `/api` sends:

```
Authorization: Bearer <user-id>
```

Resolve the current user id server-side (in the loader/action) and attach it to the
request you send to the API. Don't ship the token to the browser or hard-code it into
client components.

## Data fetching: typed operations only

Data fetching happens in route **`loader`s** (reads) and **`action`s** (writes), never
in components. Use **typed operations generated by graphql-codegen** from the backend
SDL, through the generated typed client.

- **Do not hand-roll `fetch`** to call GraphQL.
- **Do not use `as any`** (or other casts) to paper over response types. If the type
  doesn't fit, the query or the codegen is wrong — fix that.
- Keep query/mutation documents alongside the route (or in a shared operations module)
  so codegen picks them up.

```tsx
// app/routes/activity.tsx — the shape to follow (typed SDK + loader/action)
import type { Route } from "./+types/activity";
import { apiSdk } from "~/graphql/client.server";
import { getToken } from "~/sessions.server";

export async function loader({ params, request }: Route.LoaderArgs) {
  const token = await getToken(request);
  // typed query — `activity` is fully typed, no `any`
  const { activity } = await apiSdk(token).Activity({ slug: params.slug });
  if (!activity) throw new Response("Not found", { status: 404 });
  return { activity };
}

export async function action({ request }: Route.ActionArgs) {
  const token = await getToken(request);
  if (!token) return { error: "Sign in first." };

  const form = await request.formData();
  const intent = String(form.get("intent"));
  const activityId = String(form.get("activityId"));

  if (intent === "join-waiting-list") {
    try {
      // The op wraps `input` inside the .graphql document, so the SDK method takes a
      // flat `{ activityId }` — same convention as the existing RegisterToActivity op.
      const { joinWaitingList } = await apiSdk(token).JoinWaitingList({ activityId });
      return { ok: true, entry: joinWaitingList?.waitingListEntry };
    } catch {
      return { error: "Could not join the waiting list." };
    }
  }
  // ...leave-waiting-list...
}
```

> These are the **real** scaffold modules: the typed SDK is `apiSdk(token)` from
> `~/graphql/client.server`, generated types live in `~/generated/graphql`, and the
> bearer token comes from `~/sessions.server` (`getToken`). The existing register /
> unregister flow lives in a shared action helper, `registrationAction` in
> `~/lib/registration.server.ts` (intents `register` / `unregister`) — mirror it for the
> waiting list. To get the typed `JoinWaitingList` method, add a `JoinWaitingList.graphql`
> op under `app/graphql/` and run `bun run codegen`.

## Loading and error states (always)

Every interactive surface renders both. This is non-negotiable for review.

- Use `useFetcher()` (as the existing Register button does) — `fetcher.state !== "idle"`
  drives **pending UI** (disable the button, show a pending label).
- **Handle action errors** — read `fetcher.data` and render the error near the control;
  don't swallow failures. Use route **error boundaries** for thrown errors.

```tsx
import { useFetcher } from "react-router";
import { Button } from "~/components/Button"; // named export — reuse, don't restyle <button>

function WaitingListToggle({ activityId, onList }: { activityId: string; onList: boolean }) {
  const fetcher = useFetcher<typeof action>();
  const busy = fetcher.state !== "idle";

  return (
    <fetcher.Form method="post">
      <input type="hidden" name="activityId" value={activityId} />
      <input
        type="hidden"
        name="intent"
        value={onList ? "leave-waiting-list" : "join-waiting-list"}
      />
      <Button type="submit" disabled={busy}>
        {busy ? "Saving…" : onList ? "Leave waiting list" : "Join waiting list"}
      </Button>
      {fetcher.data && "error" in fetcher.data ? (
        <p className="mt-1 text-sm text-red-600" role="alert">
          {fetcher.data.error}
        </p>
      ) : null}
    </fetcher.Form>
  );
}
```

The new toggle must **mirror the existing Register button's** loading/error handling
(`~/components/RegistrationButton.tsx`, driven by the shared `registrationAction`) — match
its patterns rather than inventing a parallel one.

## Component conventions

- **Reuse existing components and helpers.** There's a shared `Button` (and friends) —
  use it instead of re-implementing one. Same for formatting/data helpers.
- **Real buttons and links only.** No clickable `<div>`s or `<span>`s. Use `<button>`,
  `<a>`, React Router `<Link>` / `<Form>` — they're keyboard- and screen-reader-
  accessible by default. An `onClick` div is an automatic review failure.
- **Stable list keys.** Lists key on a stable id (e.g. `entry.id`), **never** the array
  index.
- Type everything. Type-checking (`tsc`, no emit) is part of the gate; `any` is not an
  escape hatch.

## Quality gate (frontend)

This project uses **Bun** — use `bun`, not `npm`. There are **no pre-commit hooks** here
either. Run the gate yourself before submitting:

```bash
bun run codegen   # regenerate app/generated/graphql.ts first — check does NOT run codegen
bun run check     # biome check . && react-router typegen && tsc && vitest run
```

- **Biome** (`biome check .`) lints and checks formatting (`bun run format` applies it).
- **`tsc`** type-checks (`noEmit` comes from `tsconfig.json`), after `react-router typegen`
  refreshes route types. Run `bun run codegen` first so the generated GraphQL types exist.
- **Vitest** runs the tests.

## What "good" looks like

A reviewer scanning your frontend diff should see: typed documents through the generated
client in loaders/actions (no raw `fetch`, no `as any`); pending and error states on
every mutation; reuse of the shared `Button`; real buttons/links; stable keys; and a
green `bun run check`. The waiting-list toggle should be indistinguishable in quality
from the existing Register flow it mirrors.
